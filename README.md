# 소상공인 폐업 위험 신호 예측 및 정책 시사점 연구

2023년 MDIS 소상공인 실태조사 연간자료를 활용해, 소상공인의 **폐업 위험 신호를 조기에 탐지**하려는 머신러닝 프로젝트입니다.  
이 프로젝트는 단순히 "폐업 여부"를 맞히는 데서 끝나지 않고, **사업 지속 의향이 약화되는 패턴을 데이터로 포착하고 정책적으로 어떻게 해석할 수 있는지**까지 연결하는 데 초점을 두었습니다.

---

## 프로젝트를 시작한 이유

소상공인의 폐업은 개인의 생계 문제를 넘어 고용, 지역 상권, 정책 설계와도 연결됩니다.  
그래서 실제 폐업이 발생한 뒤를 설명하는 것보다, **위험 신호를 미리 탐지하는 모델이 더 실질적인 의미가 있다**고 생각했습니다.

특히 이 프로젝트에서는 설문 응답 데이터를 바탕으로, 사업체가 계속 운영될 가능성과 그렇지 않을 가능성을 나누고, 어떤 요인들이 그 신호를 설명하는지 살펴보았습니다.

---

## 내가 이 프로젝트에서 한 일

이 프로젝트는 단순히 모델 하나를 돌린 것이 아니라, 아래 흐름으로 진행했습니다.

1. **원자료 구조 파악**
   - MDIS 2023년 등록기반 소상공인 실태조사 데이터를 불러와 전체 변수 구조를 확인했습니다.

2. **결측치와 변수 의미 해석**
   - 단순 결측으로 보이는 값이 실제로는 "해당 없음"에 가까운 경우가 있어, 변수 의미를 먼저 해석한 뒤 전처리를 진행했습니다.

3. **피처 엔지니어링**
   - 변수명을 더 해석하기 쉬운 형태로 변경하고,
   - 의미가 섞여 있는 정책 관련 변수를 분리해 파생변수를 만들었습니다.

4. **타깃 변수 재정의**
   - 단순한 폐업 여부가 아니라, 사업전환·폐업·은퇴 계획을 포함한 **사업 지속 위험 신호**로 종속변수를 구성했습니다.

5. **탐색적 데이터 분석(EDA)**
   - 업종별 분포, 업종별 폐업 비율, 지역 분포 등을 시각적으로 확인했습니다.

6. **모델링 및 비교**
   - 클래스 불균형 문제를 고려해 baseline, undersampling, SMOTE 기반 Random Forest를 비교했습니다.
   - 이후 RandomizedSearchCV로 하이퍼파라미터 튜닝을 수행했습니다.

7. **threshold 조정**
   - 불균형 데이터에서는 정확도보다 소수 클래스 탐지가 중요하다고 보고, threshold를 바꾸며 recall과 F1-score를 비교했습니다.

8. **결과 해석**
   - confusion matrix, feature importance, SHAP를 통해 어떤 변수들이 위험 신호를 설명하는지 해석했습니다.

## 데이터

- **출처**: MDIS 「소상공인 실태조사 연간자료_등록기반」 (2023)
- **분석 대상**: 40,000개 사업체
- **원변수 수**: 156개
- **최종 사용 변수 수**: 118개

### 타깃 변수 정의
설문 문항 `사업전환_운영계획코드`를 바탕으로 종속변수를 다음처럼 재정의했습니다.

- `0`: 계속 운영
- `1`: 사업전환 / 폐업 / 은퇴 계획

즉, 이 프로젝트는 **사후 폐업 판별 모델**이라기보다, **사업 지속 의향 약화와 위험 신호를 탐지하는 모델**에 더 가깝습니다.

---

## 전처리와 피처 엔지니어링

노트북에서 실제로 수행한 주요 작업은 다음과 같습니다.

- 결측치 처리
- 재난지원, 노후준비, 세부 인원 항목 등 일부 변수 제거
- 데이터 누수를 일으킬 수 있는 변수 제거
- 문자열 변수 제거
- 상수 변수 제거
- yes/no 성격의 코드형 변수를 0/1로 재정의
- `경영_점유형태코드` → `경영_임차여부`처럼 해석 중심으로 변수명 변경
- `정부지원정책_지원경험코드`를
  - `정부지원_신청여부`
  - `정부지원_탈락여부`
  로 분리

이 과정을 통해, 단순히 모델 성능만 높이는 것이 아니라 **결과 해석이 가능한 형태의 데이터셋**을 만들고자 했습니다.

---

## 탐색적 데이터 분석

노트북에서 확인한 주요 탐색은 아래와 같습니다.

- 산업 대분류 분포
- 산업 대분류별 폐업 비율
- 산업 중분류 분포
- 지역별 분포
- 수치형 변수 중심 상관관계 탐색
- 클래스 불균형 확인

특히 `계속 운영 93.9%`, `폐업 의향 6.1%`로 불균형이 매우 크기 때문에, 이 프로젝트에서는 accuracy보다 **ROC-AUC, recall, F1-score**를 더 중요하게 봤습니다.

---

## 모델링

### 사용한 방법
- Train / Validation / Test = 60 / 20 / 20
- Stratified split 적용
- RandomForestClassifier 사용
- RandomizedSearchCV를 통한 하이퍼파라미터 탐색
- threshold 0.5 / 0.4 / 0.3 / 0.2 비교

### 비교한 접근
- Baseline Random Forest
- Undersampling Random Forest
- SMOTE Random Forest
- Tuned Random Forest

### 최종 선택
최종적으로는 **튜닝된 Random Forest + threshold 0.3** 조합을 중심 결과로 해석했습니다.

---

## 주요 결과

- **최종 모델**: Tuned Random Forest
- **Test ROC-AUC**: 0.792
- **Test F1-score**: 0.270

### 중요 변수
사후분석 결과, 다음과 같은 변수들이 중요한 설명력을 보였습니다.

- 매출
- 영업비용
- 영업이익
- 매출원가
- 부채
- 임차 관련 비용
- 영업기간
- 업종
- 지역
- 운영역량 평가 지표

### 내가 중요하게 본 해석
- 이 모델은 "폐업 확정 예측"보다는 **위험 신호 조기 탐지**에 더 적합함
- 클래스 불균형 상황에서는 높은 accuracy보다 **소수 클래스 탐지력**이 중요함
- 재무 정보뿐 아니라 운영, 입지, 정책 접근성과 같은 요소도 함께 봐야 함

## 정책적 시사점

이 프로젝트를 통해, 소상공인 지원 정책은 사후 지원보다 **사전 위험군 식별**이 더 중요할 수 있다는 점을 확인했습니다.

특히,
- 재무 상태가 악화된 사업체
- 운영 구조상 취약성이 보이는 사업체
- 정책 접근성이 낮거나 지원 과정에서 탈락한 사업체
를 조기에 식별한다면, 더 정교한 정책 설계가 가능할 수 있습니다.

## GitHub에서 작업 과정을 어떻게 보여주면 좋은가

보통 GitHub에서는 결과만 올리는 방식도 있지만, **작업 과정까지 같이 보여주고 싶다면** 아래처럼 정리하는 경우가 많습니다.

### 1. `notebooks/`
가장 흔한 영역입니다.  
EDA, 전처리, 모델링, 실험 과정을 Jupyter Notebook으로 남겨서, "어떻게 분석이 진행됐는지"를 보여줍니다.

### 2. `src/`
노트북에서 정리된 코드를 함수나 모듈 단위로 옮겨, 재사용 가능한 코드처럼 보이게 만드는 영역입니다.

### 3. `reports/` 또는 `docs/`
최종 결과 요약, 발표자료, 시각화 자료를 보관하는 곳입니다.  
다만 포트폴리오 목적이라면 보고서 전문보다 **핵심 내용 요약본**이 더 효과적입니다.

### 4. `images/` 또는 `assets/`
그래프, 모델 결과 이미지, 구조도 같은 시각 자료를 넣는 영역입니다.  
README에서 이 폴더의 이미지를 불러와 프로젝트를 더 보기 좋게 만들 수 있습니다.

즉, 사람들이 GitHub에 "작업 과정"을 같이 올린다고 할 때는 보통
- `notebooks/`
- `src/`
- `images/`
이 세 영역을 잘 써서 보여주는 경우가 많습니다.
